# teach.md — 学習用解説書（なぜ・どう動くか）

初学者（プログラミング初めて）向けに、このアプリが「なぜ・どう動くのか」を解説します。
「何があるか」を引くときは detail.md を見てください。

---

## 第0章 全体像：3つの登場人物

このアプリは「スマホ」「PC のサーバー」「PC のマウス/キーボード」の3者でできています。

```
[スマホのブラウザ]  --(指の動きを JSON で送る)-->  [PCのサーバー]  --(命令)-->  [PCのマウス/キーボード]
```

1. スマホの画面を「指でなぞる」と、その移動量を計算して
2. インターネット（同じ Wi-Fi）越しに PC のサーバーへ送り
3. サーバーがその通りにマウスを動かす

たったこれだけです。1つずつ見ていきましょう。

## 第1章 WebSocket：なぜ普通の通信じゃダメなのか

普通のウェブ（HTTP）は「ページをください」→「はいどうぞ」の**1往復で終わる**通信です。
でもマウス操作は、指を動かしている間ずっと「今こっちに動いた」「次はこっち」と
**何度も連続で**送り続けたい。毎回 HTTP で挨拶し直していたら遅くて使えません。

そこで **WebSocket** を使います。これは一度つなぐと**つなぎっぱなし**になり、
どちらからでも好きなタイミングでメッセージを送れる「電話回線」のような通信です。

- フロント側（`web/app.js`）: `new WebSocket('ws://...')` でつなぎ、`ws.send(...)` で送る。
- サーバー側（`app/server.py`）: `@app.websocket("/ws")` で受け取り、`while True` で受信し続ける。

> 💡 コラム：切れても自動で復活
> Wi-Fi が一瞬切れることはよくあります。`app.js` では `onclose` のときに
> `setTimeout(connect, 2000)` で2秒後に再接続を試みるようにしています。
> こうすると、ユーザーは「切れた」ことにほとんど気づかずに使い続けられます。

## 第2章 相対座標：カーソルを「どこへ」ではなく「どれだけ」動かす

スマホの画面と PC の画面は大きさも形も違います。だから「スマホの座標(100,200)」を
「PC の座標(100,200)」にそのまま当てはめてもズレます。

そこでトラックパッドと同じ考え方を使います。「絶対位置」ではなく
**「直前から指がどれだけ動いたか（差分）」**だけを送るのです。

```js
// app.js（イメージ）
const dx = 今の指のX - さっきの指のX;
const dy = 今の指のY - さっきの指のY;
send({ type: 'move', dx, dy });
```

サーバー側は `mouse.move(dx, dy)`（相対移動）で「今の位置から dx,dy ずらす」だけ。
これでスマホの上で指を動かした分だけ、PC のカーソルがスーッと動きます。

> 💡 コラム：感度(SENSITIVITY)
> `controller.py` の `SENSITIVITY = 1.5` は「指の動きを1.5倍にしてカーソルに伝える」設定。
> 小さな画面で大きな PC 画面を端まで動かせるよう、少し増幅しています。

## 第3章 送りすぎ防止：requestAnimationFrame で「間引く」

指を速く動かすと、`touchmove` イベントは1秒間に何十回も発生します。
そのたびに送信すると、メッセージが洪水になってカクつきます。

そこで「1画面の描画(約1/60秒)につき1回だけ送る」よう**間引き**ます。
これが `requestAnimationFrame` です。移動量をいったん変数(`accDx`,`accDy`)に
**足し込んでおき**、次の描画タイミングでまとめて1回だけ送ります。

```js
accDx += dx; accDy += dy;   // とりあえず貯める
// 次の描画のタイミングで…
send({ type:'move', dx: accDx, dy: accDy }); accDx = 0; accDy = 0;
```

## 第4章 指の本数で操作を変える（タッチイベント）

スマホは複数の指を同時に認識できます。`e.touches.length` で「今何本触れているか」が分かります。

- **1本指で動かす** → カーソル移動
- **1本指で軽くタップ**（短時間・ほぼ動かさず離す）→ 左クリック
- **2本指でタップ** → 右クリック
- **2本指で動かす** → スクロール

「タップ」かどうかは、`touchstart` から `touchend` までの**時間が短く(200ms未満)**、
**移動が小さい(10px未満)**かで判定しています（`app.js` の touchend 内）。

## 第5章 日本語入力の罠：IME と composition イベント

英語は「a」を押せば「a」が確定します。でも日本語は「k」「a」と打って「か」になり、
さらに変換して「蚊」「課」…と確定します。この**変換途中の文字**をそのまま PC に送ると、
ローマ字や未確定文字がそのまま入ってしまいます。

そこで「変換中かどうか」を `composition` イベントで見張ります。

- `compositionstart` … 変換が始まった → 送信を止める
- `compositionend` … 変換が**確定した** → そのときの確定文字列(`e.data`)だけを送る

これで「ローマ字のゴミ」を送らず、確定した日本語だけが PC に届きます。
英数字の直接入力は変換が無いので、`keydown` で1文字ずつ即送ります。

## 第6章 pynput とアクセシビリティ権限（macOS の関門）

PC 側で実際にマウス/キーを動かすのが `pynput` というライブラリ。
`mouse.move()` や `keyboard.type()` を呼ぶだけで OS にマウス/キー操作を「注入」できます。

ただし **macOS はセキュリティが厳しい**。「勝手にマウスを動かすプログラム」を OS が止めます。
これを許可するのが「**アクセシビリティ権限**」。
`システム設定 → プライバシーとセキュリティ → アクセシビリティ` でアプリを許可します。

- `permissions.py` の `is_trusted()` で権限の有無を判定（macOS のみ。Windows は権限不要で常に True）。
- 未許可なら `window.py` が「権限を設定」ボタンを出し、`open_accessibility_settings()` で
  設定画面を直接開いてあげます（苦手な人が迷わないように）。

> 💡 コラム：なぜ「アプリ自身」に権限を渡せると楽なのか
> ターミナルから動かすとターミナルに権限を与える形になり分かりにくい。
> PyInstaller で `.app` にすると「PhoneMouse.app」自身に権限を与える形になり、
> 一度許可すれば次回からそのまま動きます。

## 第7章 スレッド：サーバーと画面を同時に動かす

サーバー（待ち受け）と画面（ウィンドウ）を1本の流れで動かすと、片方が止まると
もう片方も止まります。そこで**別々の流れ（スレッド）**に分けます。

- サーバー(uvicorn) → **daemon スレッド**（裏方。メインが終われば一緒に終わる）
- ウィンドウ(tkinter) → **メインスレッド**（GUI はメインで動かすのが安全）
- トレイ(pystray) → 別スレッドで**ベストエフォート**（OS によっては出ないこともある）

`main.py` のコメントにこの方針を書いてあります。GUI 部品（tkinter/pystray）は
「メインスレッドで動かしたがる」性質があり、両立が難しいので、最優先を
「ウィンドウが出てサーバーが動いて終了できること」に置いています。

## 第8章 QR コード：IP アドレスを手入力させない工夫

スマホからつなぐには `http://192.168.x.x:8000` のような URL が必要ですが、
数字を手で打つのは間違えやすく、苦手な人にはつらい作業です。

そこで `netinfo.py` の `make_qr_image()` でこの URL を **QR コード画像**にして
ウィンドウに大きく表示します。スマホのカメラを向けるだけでブラウザが開きます。
「`get_lan_ip()`（自分の IP を調べる）→ URL を作る → QR にする」という流れです。

## 第9章 ライブ日本語入力：変換中の文字を「差分」でPCに送る

スマホで日本語を打つと、`か → かん → かんじ →（変換）→ 漢字 →（確定）`のように
文字が育っていきます。最初の版では**確定した瞬間**（`compositionend`）にだけ
「漢字」を送っていました。つまり変換中はPCに何も映りません。

「打ちながらPCに映る」ようにするには、途中経過（`compositionupdate`）も送ります。
でも毎回まるごと送って全部打ち直すと、PC側でチカチカして遅くなります。
そこで使うのが **差分（さぶん）**という考え方です。

> 💡 コラム：前方一致差分（共通の頭は送らない）
> 前回PCに送ったのが「かん」、今回が「かんじ」なら、頭の「かん」は同じ。
> 変わったのは末尾に「じ」が増えたところだけ。だから送るのは「+じ」だけでいい。
> 逆に「かんじ」→「漢字」なら、共通の頭が無いので「3文字消して『漢字』を打つ」。
> これを `{type:'compose', back:消す数, add:足す文字}` という小さな指示で送ります。

PC側（`controller.py`）は受け取った指示どおり「back 回バックスペース → add を打つ」
だけ。**自分では何も覚えません（ステートレス）**。どこまで打ったかを覚えるのは
スマホ側だけ（`composedCP`）なので、ズレる箇所が1か所に集約されて壊れにくいです。

もう一つ嬉しいのは**後方互換**。もしスマホが `compositionupdate` を出さない機種でも、
最後の `compositionend` で「全部まとめた差分」が1回出るので、昔と同じ
「確定時にまとめて入る」挙動に自然に戻ります。新機能が動かない端末でも壊れません。

> 💡 コラム：文字数の数え方の落とし穴
> 絵文字（😄 など）はコンピュータ内部では2個分の単位でできていることがあります。
> 単純な「文字数」で数えるとバックスペースの数を間違えます。そこで
> `Array.from(文字列)` を使い、絵文字も「1文字」として数えてズレを防いでいます。

## 第10章 ペアリング：合言葉(トークン)で「あなたのスマホだけ」操作できるようにする

最初の版は、同じ Wi-Fi（や Tailscale）にいる人なら**誰でも** `ws://PCのIP:8000/ws`
につなぐだけでマウスを動かせました。家の中なら問題なくても、職場や学校の
共用ネットワークでは「他人に操作される」危険があります。

そこで **トークン（合言葉）** を使います。仕組みはシンプル：

1. PC は起動するたびに**ランダムな合言葉**を1つ作る（`secrets.token_urlsafe`）。
2. その合言葉を **QR（接続URL）に埋め込む**（`http://…/?token=XXXX`）。
3. スマホは URL に入っていた合言葉を読み取り、接続時に一緒に送る
   （`ws://…/ws?token=XXXX`）。
4. PC は送られた合言葉が**自分の作ったものと一致するか**だけ確認し、
   違えば即切断。合言葉なしでは1文字も操作させません。

QR を読んだ本人だけが正しい合言葉を持っているので、QR を読んでいない他人は
（同じネットワークにいても）操作できません。合言葉は起動ごとに変わる
「使い捨て」なので、漏れても再起動すれば無効になります。

> 💡 コラム：`==` ではなく `secrets.compare_digest` を使う理由
> 合言葉の照合に普通の `==` を使うと、「最初の数文字だけ合っているか」が
> 応答時間のわずかな差から推測できてしまう攻撃（タイミング攻撃）の余地が
> 生まれます。`secrets.compare_digest` は**最後まで同じ時間で**比較するので、
> その手がかりを与えません。パスワードや鍵の比較はこれを使うのが定石です。

> 💡 コラム：起動を「固まらせない」工夫2つ
> ① Tailscale を探す処理は、見つからない環境だと数秒かかることがあります。
> これを起動の本流でやると画面が固まるので、**別スレッドで裏から**調べて
> 結果を後で受け取る形（`prime_tailscale_ip`）にしました。
> ② ブラウザで QR ページを開くのも、サーバーが準備できる前に開くと
> 「読み込み失敗」になります。そこで `/info` に**実際に問い合わせて、
> 200 が返ってきてから**開くようにしました（決め打ちの `sleep` をやめた）。
